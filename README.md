
# About the repository

This repository demonstrates the service composition language and the execution engine developed to support incremental computation in service composition. Recall that the language is based on guarded attribute grammars and asynchronous communication following a publish–subscribe scheme. Incremental computation refers to the ability of a service to process its inputs lazily, consuming only what is required at a given stage of the computation, and to produce its outputs greedily as soon as they are computed by the internal threads assigned to them. 

Incremental computation naturally enforces parallel execution: a service is not blocked while waiting for inputs that are not yet necessary, and conversely, it makes its outputs available independently as soon as they are produced. This anticipates and accelerates the execution of other services that depend on these values, resulting in a more efficient and robust overall system.

# 1 Project Structure

The project is structured as follows:

- **certificates**: This folder contains the SSL certificates automatically generated for the Docker registry inside the cluster.  
- **configs**: This folder contains the join script automatically generated by the master node to allow worker nodes to join the Kubernetes cluster.  
- **Deployment Scripts**: This folder contains the different scripts used for deploying the cluster where the services will run. It includes four files:  
  - `commons.sh`: defines the common setup required on all nodes, such as installing Kubernetes and Docker.  
  - `docker_install.sh`: a subscript used by `commons.sh` to install Docker on all nodes.  
  - `master.sh`: contains the setup specific to the master node, including kubeadm initialization and the deployment of a Docker registry used to store service images.  
  - `node.sh`: contains the setup specific to worker nodes, including trusting the SSL certificates generated by the master node and joining the Kubernetes cluster initialized by the master.  
- **GAG Service Engine**: This folder contains the Java-based engine that executes incremental services. The engine is designed to be deployed on Kubernetes using Docker, but it can also be run locally on a JVM. This allows users to test service compositions without requiring a cluster. However, this README does not cover that usage; as its purpose is to explain how to reproduce the paper results in the kubernetes cluster.  
- **Tests**: This folder contains the different test scripts as well as directories storing the output results of these tests.  

# 2 Specification 

The specification of the merge sort implemented as a composition of incremental services can be found in the folder `Tests/TestRunIn`. Recall that the principle is to represent a recursive sort with a variable number of input and output cells. In this repository, this number is defined by the parameter `$NUMBER_OF_BLOCKS`, which is configurable at deployment time.

The idea is that a recursive service `RecSort` can merge any available pair \((c_i, c_j)\), where \(i, j \in \{1, \dots, NUMBER\_OF\_BLOCKS\}\) and \(i \neq j\). In a non-incremental execution, all cells must be available before merging can begin. In contrast, incremental execution allows merging of any available pair as soon as it is ready, making the process faster and more robust.

Below is a snippet of the specification using incremental computation. The complete specification can be found in `Tests/TestRunIn/specification`. In that folder, the file `services.yml` contains the service declarations, while `rules.yml` specifies the behavior and composition rules.

```yaml
---
# Service: MainSort --- 2 rules
# R1 is active when the list is long enough to distribute computation.
kind : rule
name : MainSort_remote
service : MainSort
guard : 
 classPath: com.local.SortFunc
 method: lengthgr
 # The guard requires a minimum length l+1 of the input list
functions :
 - { classPath: com.local.SortFunc, method: merge_f, label: merge_f}
 - { classPath: com.local.SortFunc, method: getNUMBER_OF_BLOCKS, label: getNUMBER_OF_BLOCKS}
 # update_values replaces some intentional values in out_list.
actions :
 - out_array =  RecSort(inp_list)
 - length = getNUMBER_OF_BLOCKS()
 - (size, arr, out_list) =  Merge | ${NUMBER_OF_BLOCKS} (length,out_array) # merge incrementally

...

---
#Service: RecSort -- 2 rules:
# R1 is active when the list is long enough to distribute computation.
kind : rule
name : RecSort_remote
service : RecSort
guard : 
 classPath: com.local.SortFunc
 method: lengthgr_in_arr
 # The guard requires a minimum length l of the input list
functions :
 - { classPath: com.local.SortFunc, method: divideLeft, label: divideLeft}
 - { classPath: com.local.SortFunc, method: divideRight, label: divideRight}
 - { classPath: com.local.SortFunc, method: init_indices, label: init_indices}
 - { classPath: com.local.Base, method: getZero, label: getZero}
 - { classPath: com.local.Base, method: incrementNumber, label: inc}
actions :
 - pending_indices = init_indices()
 - arr_left = divideLeft(inp_arr)
 - arr_right = divideRight(inp_arr) 
 - res_left =  RecSort(arr_left)
 - res_right =  RecSort(arr_right)
 - ref_left = ~ref(res_left)
 - ref_right = ~ref(res_right)
 - index = getZero()
 - out_arr[index/pending_indices] = [ __internal Merge_S(pending_indices,ref_left,ref_right) for index,inc,${NUMBER_OF_BLOCKS}]

```

# 3 Prerequisites

To set up the cluster, you need:

- A computer with at least 16 GB of RAM and a processor with at least 8 threads.  
- Vagrant installed on your system. Installation commands depend on your OS. Ensure your version is **≥ 2.4.9**. You can check with `vagrant --version`.  
- VirtualBox installed on your system. Installation commands depend on your OS. Ensure your version is **≥ 7.0.16**. You can check with `vboxmanage --version`.  
- An operating system that supports the required versions of Vagrant and VirtualBox. We used Ubuntu 24, but any compatible OS is acceptable. Since Ubuntu is Unix-based, we recommend using a Unix-based system such as Ubuntu, Fedora, Debian, or macOS.  

# 4 Getting started

To launch the cluster, clone the repository using `git clone`. Navigate to the root folder and run `vagrant up`. You may encounter the following issues:

- **Local network error**:  
  If you encounter an error related to network creation:  
  1. Create a folder `/etc/vbox/`  
  2. Inside it, create a file `networks.conf` with the content:
     ```
     * 10.0.0.0/8 192.168.0.0/16
     * 2001::/64
     ```
  3. Run `vagrant destroy`, then `vagrant up` again.  
  This is a permanent fix.

- **VboxManage error**:  
  On Linux systems, if VirtualBox cannot operate in VMX root mode, run:
  `sudo modprobe -r kvm_intel`  
  Then rerun `vagrant up`. This fix must be applied after each reboot.

These errors are not expected but may occur in some environments. If `vagrant up` fails due to network issues, run `vagrant destroy` before retrying. Otherwise, partially created nodes may remain in an inconsistent state.

To stop the machines, use `vagrant halt`. Note that Vagrant automatically stops all machines when the host system shuts down.

## Cluster

The cluster consists of three nodes: one master and two workers. Their configurations are:
- Master: 3 vCPUs (1.80 GHz), 8192 MB RAM  
- Worker 1: 2 vCPUs (1.80 GHz), 4096 MB RAM  
- Worker 2: 2 vCPUs (1.80 GHz), 4096 MB RAM  

Kubernetes and Docker are preconfigured on all nodes. The master node runs a Docker registry secured with self-signed SSL certificates, which are trusted by all nodes.

# 5 Running the experimental tests of the scientific paper

This section explains how to reproduce the experiments presented in the paper. You must complete all previous steps first.

## Push the service image

Inside the `GAG Service Engine` folder, a Dockerfile is provided to build the image containing the engine and services.

Steps:
- Connect to the master: `vagrant ssh master`  
- Navigate to the folder:  
  `cd "/vagrant/GAG Service Engine"`  
- Run: `sudo ./start.sh`  

This builds and pushes the image to the registry.

## Run a Test

Within the `Tests` directory, there are several folders and scripts essential for managing the experimentation workflow. 

---

### Folder & Script Overview

* **`TestRunIn`**: Contains the initial specifications for the merge sort. These files use placeholders rather than static values.
    > **Example:** In `specification/deployment-worker.yml`, you will see `cpu: "${WORKER_POD_CAPACITY}m"`.
* **`TestRunOut`**: Once an experiment is launched, the placeholders are replaced with actual environment variable values, and the final specifications are stored here.
* **`changeenv.py`**: This script performs **Template Injection**. It scans `TestRunIn` for placeholders (e.g., `${WORKER_POD_CAPACITY}`) and replaces them with active environment variables to generate the deployment files in `TestRunOut`.
* **`runtest.py`**: Manages the lifecycle of a single benchmark. To ensure clean data, it uses `cpuusage.py` to verify that host CPU usage is below 15% (avoiding noise). It also performs a "warming" phase to stabilize the JVM's JIT compiler before executing tasks via `kubectl exec`.
* **`runglobaltest.py`**: This is the main entry point for running batch experiments. Environment variables can be defined in two ways:
    1.  Updating the `env_variables` dictionary directly inside the script.
    2.  Using the `--env-file` argument: `--env-file /path/to/env/file`.
* **`runglobaltestandmerge.py`**: This script allows you to re-run specific experiments that produced inconsistent results (outliers) and merge them with the initial dataset. 
    > **Pro Tip:** Processor rates can fluctuate due to thermal throttling. For consistent results, we strongly recommend running these tests only when the machine is plugged into a power source.
* **`Scripts`**: Contains subfolders (`Set1`, `Set2`, etc.) for each test set discussed in the paper. Each folder typically includes:
    * `set#.py`: Script to plot the results after experimentation.
    * `paper_data/`: The original experimental data from the paper.
    * `set#.png`: The corresponding visualization from the paper.
    * `set#.json`: The environment variables used for that specific set.
* **`Results`**: The destination folder where all experimental output is stored after a run.

> 
> Additionally, in these sets, `MAX_CONCURRENT_SERVICE_REQUEST` is restricted to `1` per pod. This simplifies failure simulation: when a failure occurs, only one data cell (an array of the threshold length) is lost. While pods still process requests in parallel, this restriction ensures that a single failure doesn't wipe out multiple concurrent tasks. We have done this simple restriction to simplify the process of creating artificial faillure. 


Each of the tests above can take up to 5 hours to complete. In particular, the first test set can take up to 8 hours to complete. You can stop a test at any time by typing `Ctrl+C` twice. Note that after you stop a step, you can resume it by running 
```bash
python3 runglobaltest.py --env-file Scripts/Set\#/set\#.json
```
, where `#` is the test number. This is particularly helpful, since you can stop a step and complete it later once you have time. However, it will resume from the last synchronization point, so you may lose the output generated in the last hour; this will be recreated automatically by the script.

Overall, this repository provides a complete and executable experimental pipeline, ensuring that all results reported in the paper can be independently reproduced and validated.
